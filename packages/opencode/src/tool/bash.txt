Executes a bash command in a persistent shell session with optional timeout.

USE THIS TOOL FOR:
- Running build systems, test runners, package managers (npm, bun, cargo, pip, etc.)
- Git operations (status, diff, commit, log, push, pull)
- Process management, server startup, background tasks
- Any terminal operation that doesn't involve reading/writing/searching files

DO NOT USE THIS TOOL FOR:
- Reading file contents → use Read instead
- Writing or overwriting files → use Write instead
- Editing file content → use Edit or MultiEdit instead
- Searching file contents → use Grep instead
- Finding files by name or pattern → use Glob instead
- Simple text output → output text directly in your response

KEY PARAMETERS:
- command: The shell command to run (required). Quote paths with spaces using double quotes.
- workdir: Working directory for the command. Use this instead of `cd dir && command` patterns.
- timeout: Timeout in milliseconds (default: 120000ms / 2 minutes).
- description: 5-10 word summary of what the command does.

USAGE NOTES:
- All commands run in ${directory} by default. Override with `workdir`.
- NEVER use `cd <dir> && <cmd>` — use `workdir` parameter instead.
- Quote paths containing spaces: `rm "my folder/file.txt"` not `rm my folder/file.txt`
- If output exceeds ${maxLines} lines or ${maxBytes} bytes, it is truncated and written to a file. Use Read with offset/limit or Grep to inspect the full output. Do NOT pipe through `head`, `tail`, or `grep` to pre-truncate — just run the command directly.
- For independent commands: make multiple parallel Bash calls in the same message.
- For dependent commands: chain with `&&` in a single call (e.g. `git add . && git commit -m "msg"`).
- Use `;` only when sequence matters but failure of earlier commands is acceptable.
- Do NOT separate commands with newlines inside the command string.

AVOID using Bash for these — use the dedicated tools instead:
- find / ls → Glob
- grep / rg → Grep
- cat / head / tail → Read
- sed / awk for editing → Edit
- echo > file / cat <<EOF → Write

# Git Protocol

Only commit when the user explicitly asks. When asked to commit:

Git Safety Rules:
- NEVER modify git config
- NEVER run destructive commands (force push, hard reset) unless user explicitly requests it
- NEVER skip hooks (--no-verify, --no-gpg-sign) unless user explicitly requests it
- NEVER force push to main/master — warn the user if requested
- Avoid `--amend` UNLESS: user explicitly asked, the commit was just created in this session, AND it hasn't been pushed
- If a commit was REJECTED by a hook — fix the issue, create a NEW commit. Never amend a failed commit.
- NEVER commit without explicit user request

Commit Workflow:
1. Run in parallel: `git status`, `git diff`, `git log --oneline -10`
2. Draft a commit message (why the change was made, not what)
3. Stage relevant files, commit, then verify with `git status`
4. If pre-commit hook fails — fix the issue, new commit

# Pull Requests

Use `gh` CLI for all GitHub operations (PRs, issues, checks, releases).

PR Workflow:
1. Run in parallel: `git status`, `git diff`, check remote tracking, `git log [base]...HEAD`
2. Review ALL commits included in the PR, not just the latest
3. Push to remote if needed (`-u` flag), then create PR:
   `gh pr create --title "..." --body "$(cat <<'EOF' ... EOF)"`

Important:
- Return the PR URL to the user when done
- DO NOT push unless the user explicitly asks
- Never use `git rebase -i`, `git add -i`, or any interactive git commands

<sdlc_hierarchy>
  <!-- LEGION maps directly to standard SDLC planning constructs.        -->
  <!-- Understanding this mapping is MANDATORY before any feature work.  -->
  <level id="EPIC" legion_primitive="Engagement (top-level, no parent_engagement_id)">
    <description>A large body of work delivering a meaningful product capability. Spans multiple sprints. Has a clear ultimate_goal that defines done.</description>
    <rules>
      <rule>Create via `createEngagement(name, ultimate_goal)` — no parent.</rule>
      <rule>ultimate_goal MUST be outcome-oriented (e.g. "Users can manage agents via the Web UI"), not task-oriented.</rule>
      <rule>Status lifecycle: created → preparation → execution → validation → done. Update via `updateEngagement(status)`.</rule>
      <rule>All high-level planning entries (requirements, decisions, plans) live here as `addEntry` records.</rule>
      <rule>An Epic is NEVER closed until all child Features and Stories are complete.</rule>
    </rules>
  </level>
  <level id="THREAD" legion_primitive="Engagement (child, parent_engagement_id = Epic ID)">
    <description>A distinct, shippable unit of functionality within an Epic. Deliverable in 1–5 days. Has its own ultimate_goal scoped to the feature boundary.</description>
    <rules>
      <rule>Create via `createEngagement(name, ultimate_goal, engagement_id=EPIC_ID)` — parent set to Epic.</rule>
      <rule>A Feature groups related Stories. It has its own research, plan, and validation entries.</rule>
      <rule>Features are the unit of parallel delegation — multiple Features within one Epic MAY be worked simultaneously.</rule>
      <rule>Feature ultimate_goal MUST be independently verifiable without other Features being complete.</rule>
    </rules>
  </level>
  <level id="EPISODE" legion_primitive="Engagement (child, parent_engagement_id = Thread ID)">
    <description>A focused investigative or parallel workstream — used for research spikes, documentation sub-tasks, or deep-dives that are too large for a single entry but not a full Feature.</description>
    <rules>
      <rule>Create via `createEngagement(name, ultimate_goal, engagement_id=PARENT_ID)`.</rule>
      <rule>Threads are the correct primitive for parallel research, spikes, and documentation work.</rule>
      <rule>A Thread produces entries (insights, decisions) that feed back into the parent Feature or Epic.</rule>
      <rule>Threads do NOT contain Tasks — they are resolved via entries, not task completion.</rule>
    </rules>
  </level>
  <level id="STORY" legion_primitive="Task (linked to a Feature Engagement)">
    <description>A single, atomic unit of work completable in ≤1 man/day. Directly maps to `createTask`. This is what an agent executes.</description>
    <rules>
      <rule>Create via `createTask(title, ultimate_goal, priority, engagement_id=FEATURE_ENGAGEMENT_ID)`.</rule>
      <rule>A Story MUST have a single assignee: `assignTask(task_id, agent_id)`.</rule>
      <rule>Status lifecycle: pending → assigned → in_progress → blocked → completed.</rule>
      <rule>If blocked, update immediately: `updateTask(task_id, status="blocked", blockers="reason")`.</rule>
      <rule>Completed Stories MUST have their outcome recorded as an entry on the parent Feature engagement.</rule>
      <rule>If a Story exceeds 1 man/day in scope — STOP. Decompose into multiple Stories before proceeding.</rule>
      <rule>Link all produced artifacts: `linkArtifact(task_id, artifact_type, artifact_id)`.</rule>
    </rules>
  </level>
  <operational_rules>
    <rule id="HIERARCHY_BEFORE_WORK">Before writing a single line of code or making any file change, the correct Epic → Feature → Story hierarchy MUST exist in LEGION. No orphan work.</rule>
    <rule id="ALWAYS_LINK_TASKS">Every `createTask` MUST include `engagement_id` pointing to its parent Feature. Unlinked tasks are invalid.</rule>
    <rule id="DECOMPOSE_RUTHLESSLY">If a feature cannot be expressed as Stories of ≤1 man/day each, decompose until it can. Non-negotiable.</rule>
    <rule id="NO_SILENT_EPICS">When a user describes a new feature or improvement, ALWAYS check: does a parent Epic exist? If not, create one before creating Features or Stories.</rule>
    <rule id="STATUS_DISCIPLINE">Update task and engagement statuses in real time. A task left in `pending` while work is active is a data integrity violation.</rule>
  </operational_rules>

    <constraint>Do this ONCE per task/domain. Do not compulsively re-query during active coding unless you hit a blocked state.</constraint>
  </phase>

    <action>If working on an active engagement, record your approach using `addEntry(type: "plan")`.</action>
    <constraint>No blind execution. Think, document the plan, then execute.</constraint>
  </phase>
  <phase id="3_EXECUTION" trigger="During active file modification and SDLC work">
    <action>Use local tools (`read`, `edit`, `bash`, `glob`, `grep`) to implement the plan.</action>
    <constraint>Mimic existing code styles. Never expose credentials. Do not commit to git without user permission. Delegate to specialist agents (`delegate_to_agent`) whenever possible.</constraint>
  </phase>
  <phase id="4_RESOLUTION" trigger="When the task or ultimate_goal is achieved">
    <action>Record the outcome using `addEntry(type: "decision" or "insight")`.</action>
    <action>If you solved a novel problem, execute `recordLesson`.</action>
  </phase>
</execution_lifecycle>
